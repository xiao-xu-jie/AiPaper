### 当前最明确的未实现清单

1. 全库语义搜索与 AI 问答。
2. 标注锚点稳定性优化。
3. 数据备份与迁移。
4. 统一下载与打包逻辑。
5. PDF 与 Markdown 精准双向定位。
6. 多论文对比与综述生成的独立工作流。
7. 图表库的收藏、插入笔记和跨论文检索。
8. 模板系统的导入导出与自动推荐。
9. 批量导入队列的文件夹递归导入、运行中取消和错误恢复。

## 产品定位建议

当前项目已经具备 PDF 上传解析、Markdown/PDF 阅读、标签管理、阅读笔记、AI 对话、翻译、高亮与注解等基础能力。

下一阶段可以把产品从“论文解析阅读器”升级为“本地论文知识库 + AI 研究助理”。重点不只是继续增加阅读工具，而是让用户能更好地检索、组织、复用自己的论文资料。

## 最值得新增的功能

### 1. 全库语义搜索与 AI 问答（未实现）

当前 AI 主要围绕当前论文工作，可以扩展为面向整个论文库的搜索和问答。

典型场景：

- 搜索“有哪些论文用了扩散模型做医学分割”
- 返回相关论文、相关段落、图表和出处
- 支持基于全库论文回答研究问题
- 支持限定标签、目录、阅读状态或时间范围

这个功能价值最高，可以把本地论文集合真正变成可查询的知识库。

### 2. 论文元数据自动提取（已实现基础版）

解析完成后自动识别并保存结构化元数据。

可提取字段：

- 标题
- 作者
- 机构
- 年份
- DOI
- arXiv ID
- 会议或期刊
- 关键词
- 摘要
- 代码仓库链接

后续可以基于这些元数据继续支持 BibTeX / RIS 导入导出、引用格式生成、Zotero 兼容等功能。

### 3. 阅读状态与研究工作流（已实现基础版）

为每篇论文增加阅读管理字段，让论文库从文件列表升级成研究工作台。

建议字段：

- 阅读状态：待读、粗读、精读、已归档、重点关注
- 评分
- 优先级
- 阅读目的
- 截止日期
- 是否已复现
- 是否已加入综述

这些字段可以和搜索、筛选、标签、AI 生成笔记联动。

### 4. PDF 与 Markdown 双向定位（暂缓）

当前应用可以在 Markdown、PDF、阅读笔记之间切换，但还没有强关联。

可考虑实现：

- 点击 Markdown 段落跳转到 PDF 对应页
- 点击 PDF 页或选区关联 Markdown 段落
- 注解同时关联 Markdown 文本与 PDF 页码
- AI 回答引用原文时带页码和段落位置

这个功能实现难度较高，但会明显提升专业阅读体验。

### 5. 批量导入与解析队列（已实现基础版）

当前已有上传能力，可以进一步增强为完整队列。

建议能力：

- 拖入文件夹批量导入 PDF
- 解析队列展示
- 暂停、继续、取消、重试
- 失败原因聚合
- 配额或网络错误的友好恢复
- 批量设置标签、目录和阅读状态

这能提升大量论文导入时的可靠性。

### 6. 多论文对比与综述生成（部分实现）

基于已有“引用论文”能力，扩展成多论文分析工具。

可生成内容：

- 研究问题对比表
- 方法对比表
- 数据集与实验指标对比
- 优缺点总结
- 贡献点归纳
- 未来工作汇总
- 综述草稿
- 组会汇报提纲

这个功能和 AI 助手的现有架构比较契合，适合作为中短期增强。

### 7. 图表库（已实现基础版）

MinerU 解析后会保存论文图片资源，可以增加独立的图表视图。

建议能力：

- 按论文查看所有图片和表格
- 自动识别图号与 caption
- 搜索 caption
- 收藏关键图
- 复制图片或 Markdown 引用
- 直接向 AI 提问某张图
- 在阅读笔记中快速插入图表

对方法图、实验图和消融表的复用很有帮助。

### 8. 模板化笔记系统（已实现基础版）

当前已有默认阅读笔记模板，可以扩展成模板管理功能。

建议模板类型：

- 快速阅读模板
- 精读模板
- 综述模板
- 复现实验模板
- 组会汇报模板
- 方法拆解模板
- 代码复现计划模板

每个模板可以绑定不同的 AI prompt，生成更稳定的结构化笔记。

## 最值得优化的功能

### 1. 标注锚点稳定性（未实现）

当前高亮和注解主要依赖 `anchorText` 恢复。遇到重复文本、Markdown 重渲染、段落结构变化时，可能定位不准。

建议改进：

- 保存段落索引
- 保存选区前后文
- 保存文本 offset
- 多策略模糊匹配
- 对重复文本显示冲突提示

这个优化能显著提升标注系统的可靠性。

### 2. 清理旧版遗留代码（已实现）

已确认根目录下的旧版 `src/app.js`、`src/agent.js`、`src/mineru.js`、`src/render.js`、`src/store.js`、`src/styles.css` 不再被当前入口引用，并已删除。

当前有效入口主要走：

- `src/main.js`
- `src/App.vue`
- `src/lib/*`
- `src/stores/*`
- `src/components/*`

后续新增或修复应继续沿 Vue 3 + Pinia 主线实现，避免重新引入旧版 DOM 控制器式代码。

### 3. 统一下载与打包逻辑（未实现）

`Viewer.vue` 和 `Sidebar.vue` 中都有 Markdown / 阅读笔记下载打包相关逻辑。

建议抽取到公共模块，例如：

- `src/lib/export.js`

可统一处理：

- 文件名清洗
- 本地图片引用提取
- Markdown + 图片资源打包
- 下载 Blob
- 错误提示格式

### 4. 统一提示体验（部分实现）

部分组件仍存在调试输出或浏览器原生提示。

建议处理：

- 移除生产环境无用的 `console.log`
- 将 `alert` 替换为统一 toast
- 统一错误文案格式
- 对网络错误、Token 错误、模型错误提供更明确的恢复建议

这会让桌面应用体验更加完整。

### 5. AI 调用体验优化（已实现基础版）

当前 AI 已支持流式输出、多模型配置和引用论文，可以继续优化交互。

建议能力：

- 取消生成
- 重试上一条
- 重新生成回答
- 编辑后重新发送
- 显示模型、耗时和 token 估算
- 复制回答时保留 Markdown
- 回答中显示引用论文来源
- 模型不可用时给出具体提示

### 6. 数据备份与迁移（未实现）

项目强调本地数据，建议增强用户对数据安全的信心。

可实现：

- 一键导出论文库索引
- 一键导出配置
- 一键恢复配置
- 检查数据目录完整性
- 修复损坏的 `meta.json`
- 迁移旧版本数据结构

### 7. 大文档性能优化（已实现基础版）

较长论文或包含大量公式、图片的 Markdown 可能带来渲染压力。

建议方向：

- Markdown 分段渲染
- 图片懒加载策略优化
- object URL 统一释放
- AI 流式 Markdown 渲染节流
- 大纲提取缓存
- 标注恢复延迟执行

## 推荐优先级

以下优先级按“当前代码已经完成基础版之后”的剩余工作重新排序。

### 第一优先级

1. 全库语义搜索与 AI 问答。
2. 标注锚点稳定性优化。
3. 数据备份与迁移。
4. 统一下载与打包逻辑。

这几项分别对应知识库核心价值、阅读数据可靠性、本地数据安全和维护成本，建议优先处理。

### 第二优先级

1. 多论文对比与综述生成独立工作流。
2. 图表库增强：收藏、插入笔记、跨论文检索。
3. 批量导入增强：文件夹递归导入、运行中取消、错误恢复。
4. 统一提示体验：全量 toast、错误恢复建议、清理调试输出。

这些功能能继续放大已有元数据、工作流、模板和图表库的价值。

### 第三优先级

1. 清理旧版遗留代码。
2. 模板系统增强：导入导出、自动推荐、版本管理。
3. 大文档性能进一步优化：分段/虚拟渲染、大纲缓存。
4. PDF 与 Markdown 精准双向定位。

这些可以穿插推进；其中 PDF 与 Markdown 双向定位难度较高，建议等有更可靠的页码/段落映射方案后再重启。

## 建议的演进路线

比较稳妥的路线是：

1. 先做全库语义搜索的最小闭环：建立段落索引、返回论文和片段、让 AI 回答带出处。
2. 同步补标注锚点稳定性和数据备份，保证用户积累的数据更可靠。
3. 抽取下载打包公共模块，顺手推进旧代码清理和统一提示体验。
4. 在全库检索可用后，再扩展多论文对比、综述生成和跨论文图表检索。
5. 等基础数据和索引更稳定，再评估是否重启 PDF 与 Markdown 双向定位。

这样每一步都能复用现有架构，不需要推翻重做。
